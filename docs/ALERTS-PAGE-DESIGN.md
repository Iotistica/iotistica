# Alerts Page - Design & Implementation Plan

## Overview
A dedicated page for viewing, managing, and acknowledging anomaly detection alerts generated by the edge-based anomaly detection system.

## ⚡ CRITICAL: Database Schema Update Required

**Migration**: `api/database/migrations/134_anomaly_endpoint_tracking.sql` (auto-runs on API build)

The original anomaly schema tracked **agent UUIDs** in the old `device_id` field, which is infrastructure info users don't care about. The schema has been updated to track **monitored devices** (e.g., "COMAP-Main-Controller", "Temp-Sensor-01") instead.

**Key Changes**:
- Old `device_id` → `agent_uuid` (infrastructure tracking)
- Added `device_name` (the actual monitored device users care about)
- Added `device_type` (modbus, opcua, bacnet, mqtt-sensor, agent-system)
- `affected_devices` column retained (already correct)

**To Apply**: Rebuild API container: `docker compose up -d api --build`

**Impact**: Agent code must be updated to send `deviceName` and `deviceType` in anomaly MQTT events.

## Backend Status (What's Already Implemented)

### Database Schema (110_anomaly_correlation.sql → 134_anomaly_endpoint_tracking.sql)

#### 1. `anomaly_events` Table (TimescaleDB Hypertable)
- **Purpose**: High-volume time-series storage of raw anomaly events from edge devices
- **Partitioning**: 1-day chunks on `timestamp_ms` (integer milliseconds)
- **Compression**: Automatic after 14 days
- **Retention**: 90 days automatic cleanup
- **Key Fields**:
  - `msg_id` - Unique message identifier
  - `agent_uuid` - Edge gateway/agent UUID (infrastructure tracking)
  - `device_name` - **⚡ CRITICAL**: Monitored device name (e.g., 'COMAP-Main-Controller', 'Temp-Sensor-01')
  - `device_type` - Source type: 'modbus' | 'opcua' | 'bacnet' | 'mqtt-sensor' | 'agent-system'
  - `metric` - Metric name (e.g., 'temperature', 'cpu_usage', 'ActivePower')
  - `timestamp_ms` - Event timestamp (milliseconds)
  - `observed_value` - Actual measured value
  - `anomaly_score` - 0.0-1.0 (higher = more anomalous)
  - `confidence` - 0.0-1.0 (detection confidence)
  - `severity` - 'info' | 'warning' | 'critical'
  - `fingerprint` - Hash for correlation (device+metric+method+severity)
  - `triggered_by` - JSONB array of detection methods
  - `baseline` - JSONB with statistical baseline
  - `expected_range` - JSONB [min, max]
  - `deviation` - How far from normal

**⚠️ IMPORTANT CHANGE**: The schema was updated to track **monitored devices** instead of just agent UUIDs. Users care about "Temperature Sensor 01" having an anomaly, not "agent-abc123".

#### 2. `anomaly_incidents` Table (Regular Table)
- **Purpose**: Low-volume aggregated incidents (correlated events by fingerprint)
- **Key Fields**:
  - `incident_id` - UUID for incident
  - `fingerprint` - Correlation key (device+metric+method)
  - `device_name` - Primary monitored device name
  - `device_type` - Device type
  - `metric` - Affected metric
  - `severity` - 'info' | 'warning' | 'critical'
  - `affected_devices` - JSONB array of device names (if multi-device incident)
  - `affected_agents` - JSONB array of agent UUIDs (infrastructure tracking)
  - `first_seen` - First event timestamp (ms)
  - `last_seen` - Last event timestamp (ms)
  - `max_anomaly_score` - Highest score in incident
  - `max_confidence` - Highest confidence
  - `event_count` - Number of correlated events
  - `status` - 'open' | 'active' | 'resolved'

**⚡ KEY CHANGE**: Added `device_name` and `device_type` to track actual monitored devices, keeping `affected_devices` as-is.

#### 3. `anomaly_alerts` Table (Regular Table)
- **Purpose**: Alert notifications triggered from incidents
- **Key Fields**:
  - `alert_id` - UUID
  - `incident_id` - References anomaly_incidents
  - `severity` - 'info' | 'warning' | 'critical'
  - `device_name` - Primary monitored device
  - `metric` - Metric name
  - `affected_devices` - JSONB array of device names
  - `max_anomaly_score` - Peak score
  - `message` - Human-readable message
  - `channels` - JSONB routing metadata
  - `created_at` - Timestamp

### MQTT Handler (anomaly-handler.ts)
- ✅ Receives anomaly events via MQTT from edge devices
- ✅ Deduplicates by fingerprint
- ✅ Correlates events into incidents (Redis + PostgreSQL)
- ✅ Severity escalation logic
- ✅ Alert triggering (stored to `anomaly_alerts` table)
- ✅ Cross-device correlation

### Existing API Routes (anomaly.ts)
Current endpoints focus on anomaly **scores** and **aggregates**, NOT incidents/alerts:
- `GET /api/anomaly/summary` - Device anomaly summary
- `GET /api/anomaly/hourly` - Hourly aggregates
- `GET /api/anomaly/daily` - Daily aggregates
- `GET /api/anomaly/top-metrics` - Top anomalous metrics

**Missing**: No endpoints for querying `anomaly_incidents` or `anomaly_alerts` tables!

The agent must include monitored device info when publishing anomaly events:

**Current (Broken)**:
```typescript

  // ...
```

**Fixed**:
```typescript

const event: AnomalyEvent = {
  agentUuid: process.env.AGENT_UUID,           // Infrastructure tracking
  deviceName: 'COMAP-Main-Controller',         // ✅ What users care about
  deviceType: 'modbus',                        // Source type
  metric: 'ActivePower',
  observedValue: 85.3,
  // ...
  fingerprint: hashFingerprint({               // Include device in fingerprint
    device: 'COMAP-Main-Controller',
    metric: 'ActivePower',
    method: 'zscore',
    severity: 'critical'
  })
```


### 4. Custom Components Needed

#### A. SeverityBadge Component
```tsx
function SeverityBadge({ severity }: { severity: 'info' | 'warning' | 'critical' }) {
  const variants = {
    info: { variant: 'default', icon: Info, className: 'bg-blue-500' },
    warning: { variant: 'warning', icon: AlertTriangle, className: 'bg-yellow-500' },
    critical: { variant: 'destructive', icon: AlertOctagon, className: 'bg-red-500' }
  };
  
  const { icon: Icon, className } = variants[severity];
  
  return (
    <Badge className={className}>
      <Icon className="w-3 h-3 mr-1" />
      {severity.toUpperCase()}
    </Badge>
  );
}
```

#### B. StatusBadge Component
```tsx
function StatusBadge({ status }: { status: 'open' | 'active' | 'resolved' }) {

    resolved: { variant: 'success', className: 'bg-green-500' }

  const { className } = variants[status];
  
  return (

      {status.toUpperCase()}
    </Badge>

```

#### C. ScoreBadge Component
```tsx
function ScoreBadge({ score }: { score: number }) {
  const getColor = () => {
    if (score >= 0.85) return 'bg-red-500';
    if (score >= 0.7) return 'bg-yellow-500';
    return 'bg-blue-500';
  };
  
  return (
    <Badge className={getColor()}>
      {score.toFixed(3)}
    </Badge>
  );
}
```

## Notes & Considerations

1. **TimescaleDB Performance**: The `anomaly_events` table can grow very large. Make sure queries use indexes and chunk exclusion effectively.

2. **Redis Caching**: The MQTT handler uses Redis for incident correlation. Ensure Redis connectivity is stable.

3. **Real-Time Updates**: Consider WebSocket subscription to `/api/ws/anomaly-incidents` for live updates without polling.

4. **Mobile Responsiveness**: Use responsive grid layouts and collapse filters on mobile.

5. **Accessibility**: Ensure proper ARIA labels, keyboard navigation, and screen reader support.

6. **Permissions**: Add role-based access control (RBAC) if needed - who can resolve incidents?

7. **Audit Trail**: Consider logging all incident resolutions for compliance.

8. **Export Functionality**: Users may want to export incident reports to CSV/PDF for analysis.
